# Subtheme anatomy

After `drush radix:create SUBTHEME_NAME`, your new directory at `web/themes/custom/SUBTHEME_NAME/` looks roughly like this. This page explains every file and folder so you know what to edit and what to leave alone.

## Tree at a glance

```
SUBTHEME_NAME/
├── SUBTHEME_NAME.info.yml          # theme metadata — base theme, regions, libraries
├── SUBTHEME_NAME.libraries.yml     # CSS/JS library definitions
├── SUBTHEME_NAME.theme             # PHP — loads includes/*.theme
├── SUBTHEME_NAME.breakpoints.yml   # responsive breakpoints for Drupal's image system
├── webpack.mix.js                  # Laravel Mix asset pipeline
├── package.json                    # npm dependencies + scripts
├── package-lock.json               # auto-generated, do not hand-edit
├── biome.json                      # BiomeJS lint/format config
├── mix-manifest.json               # auto-generated by Mix, do not edit
├── .env.example                    # template for .env.local
├── .nvmrc                          # pinned Node version
├── README.md                       # subtheme docs
├── logo.svg                        # site logo (replace with your own)
├── screenshot.png                  # shown in Appearance UI
│
├── src/                            # source assets — what you edit
│   ├── assets/                     # fonts, images, icons, video
│   │   ├── fonts/
│   │   ├── images/
│   │   └── icons/
│   ├── js/                         # global JS
│   │   ├── _bootstrap.js           # imports the Bootstrap JS bundles you need
│   │   └── main.scripts.js         # entry point for global behaviors
│   └── scss/                       # SCSS source
│       ├── style.scss              # main entry — imports everything else
│       ├── _variables.scss         # YOUR Bootstrap variable overrides
│       ├── base/                   # base layer (typography, resets, etc.)
│       └── components/             # optional global component imports
│
├── build/                          # compiled output — gitignore in CI
│   ├── css/
│   ├── js/
│   └── assets/
│
├── components/                     # YOUR SDC components — start empty, copy here to override
│   └── offcanvas/                  # one shipped example so the folder isn't empty
│       ├── offcanvas.twig
│       ├── offcanvas.component.yml
│       ├── offcanvas.scss
│       ├── _offcanvas.js
│       └── README.md
│
├── templates/                      # Twig template overrides
│   ├── page.html.twig              # main page layout
│   ├── node/
│   ├── block/
│   ├── form/
│   ├── field/
│   ├── views/
│   ├── navigation/
│   └── …
│
├── includes/                       # broken-out .theme files, auto-loaded
│   └── *.theme                     # one per concern (e.g. preprocess_node.theme)
│
├── config/                         # config YAML (often empty)
│   ├── install/
│   ├── optional/
│   └── schema/
│
├── translations/                   # .po language files
│   └── nb.po                       # example: Norwegian
│
└── node_modules/                   # npm — auto-generated, gitignore
```

## Root-level files

### `SUBTHEME_NAME.info.yml`

Drupal theme metadata. Declares the base theme, regions, libraries, and SDC namespace.

```yaml
name: SUBTHEME_NAME
type: theme
base theme: radix
description: 'A Radix-based subtheme for ...'
package: Custom
core_version_requirement: ^10.1 || ^11

# Libraries loaded on every page
libraries:
  - SUBTHEME_NAME/global-styling
  - SUBTHEME_NAME/global-scripts

# Block regions exposed in Block Layout
regions:
  navbar: 'Navbar'
  header: 'Header'
  highlighted: 'Highlighted'
  help: 'Help'
  content: 'Content'
  sidebar_first: 'Sidebar first'
  sidebar_second: 'Sidebar second'
  footer: 'Footer'
  page_top: 'Page top'
  page_bottom: 'Page bottom'

# SDC namespace registration — automatic on Drupal 10.3+, but explicit doesn't hurt
components:
  namespaces:
    SUBTHEME_NAME:
      - components
```

**Edit this when:** adding/removing regions, adding new global libraries, changing the displayed theme name.

### `SUBTHEME_NAME.libraries.yml`

Defines CSS/JS libraries.

```yaml
global-styling:
  version: VERSION
  css:
    theme:
      build/css/style.css: { minified: false }
  dependencies:
    - core/drupal

global-scripts:
  version: VERSION
  js:
    build/js/main.scripts.js: { minified: false }
  dependencies:
    - core/drupal
    - core/once
```

**Edit this when:** adding a new library (e.g. a third-party plugin), attaching different stylesheets to different contexts, or changing dependencies. Component libraries (loaded via SDC) don't go here — they're declared in `component.yml` `libraryOverrides`.

### `SUBTHEME_NAME.theme`

PHP file loaded by Drupal. The Radix-shipped one is just a loader:

```php
<?php
/**
 * @file
 * Functions to support theming in the SUBTHEME_NAME theme.
 */

// Load all .theme files in includes/
$includes = glob(__DIR__ . '/includes/*.theme');
foreach ($includes as $include) {
  require_once $include;
}
```

**Don't put logic here.** Add a file in `includes/` instead. Common ones: `includes/preprocess.theme`, `includes/suggestions.theme`, `includes/forms.theme`.

### `SUBTHEME_NAME.breakpoints.yml`

Drupal's breakpoint definitions, used by responsive image styles and breakpoint-aware blocks.

```yaml
SUBTHEME_NAME.mobile:
  label: mobile
  mediaQuery: 'all and (max-width: 767px)'
  weight: 0
  multipliers:
    - 1x

SUBTHEME_NAME.tablet:
  label: tablet
  mediaQuery: 'all and (min-width: 768px) and (max-width: 991px)'
  weight: 1
  multipliers:
    - 1x
    - 2x

SUBTHEME_NAME.desktop:
  label: desktop
  mediaQuery: 'all and (min-width: 992px)'
  weight: 2
  multipliers:
    - 1x
    - 2x
```

**Edit this when:** Drupal's responsive image module needs different breakpoints than Bootstrap's grid. CSS media queries belong in SCSS, not here.

### `webpack.mix.js`

Laravel Mix asset pipeline. See `setup-and-build.md` → *webpack.mix.js* for what it does and the rare cases when you edit it.

### `package.json`

npm dependencies and scripts. Add new packages as `devDependency`:

```bash
npm install --save-dev some-package
```

Don't hand-edit `package-lock.json`.

### `biome.json`

BiomeJS lint + format config. Shipped defaults are good. Edit only if you have a real reason (e.g. tab vs. spaces, line width).

### `.env.example` / `.env.local`

`.env.example` is the committed template. Copy to `.env.local` (gitignored) and set `DRUPAL_BASE_URL` for BrowserSync.

### `mix-manifest.json`

Auto-generated by Mix after each build. Maps source filenames to hashed output names when `mix.version()` is on. Never edit by hand.

## `src/` — source assets

What you actually edit during development.

### `src/scss/`

- `style.scss` — main entry. Imports `_variables.scss`, Bootstrap, your base styles, component overrides.
- `_variables.scss` — your Bootstrap variable overrides go here, **before** importing Bootstrap. See `bootstrap-bootswatch.md`.
- `base/` — typography, resets, body-level styles, etc.
- `components/` — global component CSS that doesn't belong to one SDC component.

### `src/js/`

- `_bootstrap.js` — imports the Bootstrap JS modules you actually use. Tree-shake by commenting out the unused ones.
- `main.scripts.js` — entry point. Wrap your behaviors in Drupal's `Drupal.behaviors` pattern:

```javascript
((Drupal, once) => {
  Drupal.behaviors.subthemeName = {
    attach(context) {
      once('subthemeName', '[data-subthemeName]', context).forEach((el) => {
        // your code
      });
    },
  };
})(Drupal, once);
```

### `src/assets/`

Fonts, images, icons, video. The Mix config copies these to `build/assets/`. Reference them in CSS as `../assets/...` (relative from `build/css/`).

## `components/` — your SDC components

Starts with one shipped example (`offcanvas/`). Add components here:

- **To override a Radix-shipped component:** copy `web/themes/contrib/radix/components/<name>/` into here. SDC will use yours automatically. See `components-guide.md` → *Overriding*.
- **To create a new custom component:** `npx drupal-radix-cli generate` or copy `assets/component-template/` from this skill. Component name = folder name = include name. See `components-guide.md` → *Creating components*.

## `templates/` — Twig template overrides

Drupal's core/contrib templates that you've overridden. Discover the right filename via Twig debug comments (`* node--article.html.twig`, etc.). Drop the matching `<name>.html.twig` into the appropriate subfolder.

Conventional subfolders Radix creates:

- `block/`, `field/`, `form/`, `navigation/`, `node/`, `taxonomy/`, `user/`, `views/`

Each contains starter overrides that already include the right `radix:*` SDC component. Edit those instead of writing fresh markup.

## `includes/` — broken-out .theme files

Auto-loaded by `SUBTHEME_NAME.theme`. Organize by concern:

```
includes/
├── preprocess.theme              # hook_preprocess_*
├── suggestions.theme             # hook_theme_suggestions_*_alter
├── form_alter.theme              # hook_form_*_alter
└── ...
```

Each file is plain PHP. Functions inside follow the standard `THEMENAME_*` naming.

## `config/`

Mostly empty. Config YAML you want exported with the theme (rare) goes here.

## `translations/`

`.po` files for theme strings. Drupal's locale system picks them up automatically.

## `node_modules/` and `build/`

Auto-generated. Gitignore both in CI environments. For deploy: either commit `build/` or run `npm ci && npm run production` in the pipeline.

## What you should almost never edit

- `mix-manifest.json` — auto-generated
- `package-lock.json` — auto-generated
- `screenshot.png`, `logo.svg` — replace, don't edit
- Anything in `web/themes/contrib/radix/` — that's the parent. Subtheme overrides go in *your* subtheme.
